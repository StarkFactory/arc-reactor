## Cycle 4 상세 (R270~R290)

**Before**: 백그라운드 코루틴이 자기 스코프를 정리하지 못하고 누적되었고, 일부 맵은 무제한으로 성장했으며,
스냅샷 TOCTOU 경쟁으로 잘못된 데이터를 반환하는 경로가 있었습니다.

**After**:

### 1. ConcurrentHashMap → Caffeine bounded cache (7 클래스, 17 필드)

| 클래스 | 마이그레이션한 필드 수 |
|---|---|
| `McpManager` | 4 |
| `McpHealthPinger` | 1 |
| `AgentMetrics` + `MicrometerAgentMetrics` | 10 |

장시간 실행 시 OOM 위험을 제거하고 `maximumSize`로 soft upper bound 적용.

### 2. Race condition / TOCTOU 9건 수정

- `ToolCallOrchestrator.findMcpToolCallback` TOCTOU 제거 (R273)
- `JdbcPersonaStore`에 `SELECT FOR UPDATE` 추가 (R284)
- `InMemoryPersonaStore.delete()` 동기화 + 200회 stress 회귀 (R285)
- `MicrometerSlaMetrics` availability counter atomicity (R286)
- `McpConnectionSupport`에 `throwIfCancellation()` 추가 → 코루틴 취소 제대로 전파 (R278)
- `McpHealthPinger` SupervisorJob 미취소 lifecycle leak 수정 (R283)
- 기타 `ConversationManager` 뮤텍스, `StreamingReActLoopExecutor` multi-chunk toolCall 누적 수정 등

### 3. 보안 강화 (R288)

- `TokenRevocationStore`가 Redis 연결 실패 시 조용히 in-memory로 폴백하던 경로를 **strict 모드**로 전환.
  이제 명시적으로 fail-fast하며 인증 시스템이 절반-망가진 상태로 돌아가지 않음.

### 4. Cycle 4 종결 (R290)

- `ForkJoinPool.commonPool`을 쓰던 startup probe를 전용 daemon executor로 분리.
- 앱 기동 작업이 공용 풀과 격리되어 서로 방해하지 않음.

## Cycle 5 상세 (R291~R304) — 가장 큰 변화

**Before**: `arc-web`과 `arc-admin`이 WebFlux(Reactor Netty) 위에서 돌면서 JDBC 블로킹 쿼리를

다른 요청들도 같이 느려지거나 타임아웃나는 위험한 구조였습니다.

**After**:

### 1. arc-admin 11/11 컨트롤러 전부 suspend + `withContext(Dispatchers.IO)` 격리 (R301~R304, 4 라운드)

| 배치 | 라운드 | 컨트롤러 |
|---|---|---|
| 1차 | R301 | SlackActivity, ToolCall, EvalDashboard (3개) |
| 2차 | R302 | Latency, TokenCost, Usage, ConversationAnalytics (4개) |
| 3차 part 1 | R303 | Trace, TenantAdmin (2개, 12 핸들러) |
| 3차 part 2 | R304 | **Platform** (1개, 659 lines, 20 핸들러 중 18개) |

총 **11/11 컨트롤러, 약 50+ 핸들러** 변환. 이제 admin 대시보드가 WebFlux 이벤트 루프를 블로킹하지 않음.

### 2. arc-web blocking 호출 정리 (R293~R297, 5건)

## Cycle 6 상세 (R305~R312, 진행 중)

스캐너 재투입 + 직접 grep 조사로 라우팅, 플래너, Approval, 멀티에이전트, 스케줄러, 메모리 영역을 탐색.

### Intent / Routing 영역 (R305~R307, 100% 종결)

1. **R305 P1 #1** — `IntentConfiguration`이 `arc.reactor.intent.enabled=true` + DataSource 없음 환경에서 `NoSuchBeanDefinitionException`으로 기동 실패하던 silent startup failure 수정. `@ConditionalOnMissingBean(IntentRegistry::class)` 폴백 빈 추가 → DB 없어도 `InMemoryIntentRegistry`로 내려감.

2. **R305 P1 #2** — `InMemoryIntentRegistry`의 `ConcurrentHashMap`을 Caffeine bounded cache로 전환 (`maxEntries=10_000`). 동적 intent 등록 시 OOM 방지.

3. **R306 P2 TOCTOU** — `JdbcIntentRegistry`의 `@Volatile var snapshot` double-check locking에서 `invalidateSnapshot()`이 `synchronized` 밖에서 `snapshot = null`을 쓰던 race를 `AtomicReference`로 전환.

4. **R306 P2 regex validation** — `ToolRoutingConfigValidator`에 `regexPatternRef` 화이트리스트 검증 추가. 전에는 tool-routing.yml 오타가 첫 요청 시 크래시, 지금은 기동 시 fail-fast.

5. **R307 P3 log sanitize** — `LlmIntentClassifier` 파싱 실패 시 LLM raw 응답 전체를 WARN 로깅하던 정보 유출 경로 차단. `sanitizeForLog()` helper로 첫 100자 + 전체 길이만 출력.

### Planner 영역 (R308~R309, 100% 종결)

6. **R308 #1 (P1)** — `PlanExecuteStrategy.synthesize()`와 `directAnswer()`가 LLM null/빈 응답을 `AgentResult.success("")`로 silent 반환하던 버그를 `INVALID_RESPONSE` 실패 승격.

7. **R308 #2 (P2)** — 모든 plan step이 실패했을 때 synthesize LLM을 호출해 환각 응답을 생성하던 버그를 `results.isNotEmpty() && results.none { it.success }` 조건으로 우회, `TOOL_ERROR` 반환. **narrow 조건**으로 `maxToolCalls=0` edge case는 기존 동작 유지.

8. **R308 #3 (P3)** — tool 예외의 `e.javaClass.simpleName`이 synthesis 경유 사용자 응답에 노출되던 경로를 불투명 `"Error: TOOL_ERROR"` 고정 문자열로 차단.

9. **R309 P2** — `PlanExecuteStrategy`의 `generatePlan`/`synthesize`/`directAnswer` 3개 LLM 호출이 `recordTokenUsage`를 호출하지 않아 **PLAN_EXECUTE 모드 토큰 비용이 메트릭 파이프라인에서 누락**되던 버그 수정. constructor에 파라미터 추가하고 `SpringAiAgentExecutor`에서 `agentMetrics.recordTokenUsage` 람다 wiring.

### Approval 영역 (R310)

10. **R310 P2** — `InMemoryPendingApprovalStore`의 Human-in-the-Loop 승인 저장소 `pending` 맵이 unbounded CHM이라 대량 제출 시 OOM 위험. Caffeine bounded cache(`maxPending=10_000`)로 전환. Eviction 시에도 `CompletableDeferred`가 메모리에 여전히 존재하므로 `withTimeoutOrNull`이 자연스럽게 타임아웃 처리 → coroutine leak 없음.

**+ arc-app 탐색 종결**: R305/R308에서 스캐너가 두 번 부분 반환한 원인이 "arc-app은 .kt 소스 0개의 Spring Boot assembly 껍질 모듈"이라는 사실로 판명. 향후 스캔 대상 제외.

### Multiagent 영역 (R311)

11~13. **R311** — multiagent 디렉토리 3개 파일의 독립적 CHM을 동일 Caffeine 패턴으로 전환:
- `DefaultAgentRegistry.agents` (`maxAgents=1000`)
- `DefaultSharedAgentContext.store` (`maxEntries=10_000`)
- `InMemoryAgentMessageBus.subscribers` (`maxSubscribers=1000`) — nested CoWAL을 Caffeine `get(key, mappingFunction)` atomic get-or-create로 race-free 유지

### Scheduler / Memory Summary / User Memory 영역 (R312 — 이번 라운드)

14~16. **R312** — InMemory 스토어 3종 병렬 Caffeine 전환:
- `InMemoryScheduledJobStore.jobs` (`maxJobs=1000`)
- `InMemoryConversationSummaryStore.store` (`maxSummaries=10_000`)
- `InMemoryUserMemoryStore.store` (`maxUsers=10_000`) — `compute()` atomic upsert는 Caffeine `asMap()` ConcurrentMap 뷰로 위임하여 원자성 유지

## 지금 쓰는 사람 입장에서 체감될 변화

1. **더 안정적**: 장시간 운영해도 메모리가 무제한 성장하지 않고, MCP 재연결이 깨끗하며, 동시 요청이 서로 방해하지 않음
2. **더 빠름**: arc-admin 대시보드가 WebFlux 이벤트 루프를 블로킹하지 않아 부하 상황에서 응답성이 크게 개선됨
3. **더 안전**: 관리자 예외 메시지 노출, 익명 사용자 bypass, prompt injection 로그 유출 등 정보 노출 경로 8건 차단
4. **더 관찰 가능**: PLAN_EXECUTE 모드 토큰 비용이 메트릭에 반영되고, 로그에 예외가 남아도 사용자 응답엔 안 섞임
5. **기동 신뢰성 향상**: Intent 시스템, 라우팅 validator 등에서 silent failure가 fail-fast로 전환되어 잘못된 설정이 배포 타임에 잡힘

**기능적으로는 사용자가 쓰는 모든 기존 기능이 동일하게 작동합니다.** 오히려 silent failure나 blocking으로 불안정했던 경로들이 이제 예측 가능한 동작을 합니다.
