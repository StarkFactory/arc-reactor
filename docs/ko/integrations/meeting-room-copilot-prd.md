# 회의실 코파일럿 PRD 초안

## 1. 개요

회의실 코파일럿은 다우오피스의 회의실 예약 정보와 Slack 사용자 정보를 연결해,
직원이 회의실을 더 빨리 찾고, 더 적절한 공간을 고르고, 필요하면 바로 예약자에게
연락할 수 있게 돕는 기능이다.

이 문서는 Arc Reactor 위에 회의실 코파일럿 기능을 붙이기 위한 제품 요구사항 초안이다.

## 2. 문제 정의

현재 회의실 사용 경험에는 아래 문제가 있다.

- 지금 바로 쓸 수 있는 회의실을 빠르게 찾기 어렵다.
- 이번 주 안에 가능한 시간과 회의실을 함께 찾기 어렵다.
- 인원수에 맞는 회의실을 고르기 어렵다.
- 회의실 이름만 알아서는 몇 층 몇 호인지 바로 알기 어렵다.
- 다우오피스에는 예약자 이름만 보여 누군지 식별이 어렵고 바로 연락하기 불편하다.
- 예약만 해두고 실제로 사용하지 않는 회의실이 발생한다.

## 3. 목표

이 기능의 목표는 아래 두 가지다.

1. 회의실 탐색과 선택 시간을 줄인다.
2. 안 쓰는 예약을 줄여 실제 가용 회의실을 늘린다.

## 4. 핵심 사용자 질문

사용자는 자연어로 아래 질문을 할 수 있어야 한다.

- 지금 회의하고 싶은데 어디 비지?
- 이번 주 회의해야 하는데 언제 어디가 비지?
- 우리 4명인데 어디 회의실 가야 되지?
- 이 회의실 몇 층 몇 호지?
- 나 다음 회의 있는데 회의실 어디지?
- 지금 드림룸 누가 쓰고 있어?

## 5. 사용자 가치

### 5.1 지금 바로 쓸 수 있는 회의실 찾기

사용자는 현재 시점 기준으로 바로 들어가 쓸 수 있는 회의실을 빠르게 찾을 수 있어야 한다.

제공 정보:

- 현재 비어 있는 회의실 목록
- 각 회의실의 다음 예약 시작 시각
- 수용 인원
- 층/호수

### 5.2 시간대 기준 회의실 찾기

사용자는 오늘 또는 이번 주 기준으로 가능한 시간과 회의실을 함께 찾을 수 있어야 한다.

제공 정보:

- 날짜/시간 조건에 맞는 빈 슬롯
- 해당 시간에 사용 가능한 회의실
- 인원 조건에 맞는 후보만 노출

### 5.3 인원수에 맞는 회의실 추천

사용자는 참석 인원에 맞는 "적절한" 회의실을 추천받을 수 있어야 한다.

기본 추천 기준:

- 인원 수용 가능
- 요청한 시간대 사용 가능
- 너무 큰 방보다 적정 크기 우선
- 현재 비어 있으면 우선 추천

### 5.4 회의실 위치 안내

사용자는 회의실 이름만으로 위치를 바로 확인할 수 있어야 한다.

제공 정보:

- 몇 층 몇 호
- 수용 인원
- 주요 메타데이터가 있으면 함께 제공
- 예: 화상회의 가능 여부, TV/모니터 여부

### 5.5 내 다음 회의 안내

사용자는 "나 다음 회의 있는데 회의실 어디지?"라고 물으면 본인 일정 기준 다음 회의를 확인할 수 있어야 한다.

전제:

- 해당 회의가 다우오피스 캘린더에 등록되어 있어야 한다.
- 회의실 정보가 캘린더 이벤트에 연결되어 있어야 한다.

제공 정보:

- 다음 회의 제목
- 시작 시각
- 회의실 이름
- 층/호수

### 5.6 예약자 식별 및 바로 연락

사용자는 특정 회의실의 현재 예약자를 확인하고, Slack에서 바로 연락할 수 있어야 한다.

제공 정보:

- 다우오피스 예약자 이름
- Slack 표시 이름
- Slack 프로필 링크 또는 DM 진입 링크
- 필요 시 부서 정보

주의:

- v1에서 "사용 중"은 실제 점유가 아니라 "현재 예약 중인 사람" 기준으로 제공한다.
- 이후 확인 알림 응답이 있으면 "사용 확인됨" 상태로 구분한다.

## 6. MVP 범위

MVP에는 아래 기능을 포함한다.

- 현재 비어 있는 회의실 조회
- 오늘/이번 주 가능한 회의실 시간 조회
- 인원 수 기준 회의실 추천
- 회의실 층/호수 조회
- 내 다음 회의실 조회
- 현재 예약자 확인
- Slack 프로필 연결 및 바로 DM 가능한 정보 제공

## 7. 후속 기능

MVP 이후 바로 붙일 수 있는 기능은 아래다.

- 회의 시작 5분 전 예약자에게 사용 여부 확인 알림 발송
- 응답 버튼 제공
  - 사용 예정
  - 사용 안 함
  - 나중에 다시 확인
- 사용 안 함 응답 시 예약 해제 지원

기본 정책:

- Slack DM이 가능하면 Slack으로 먼저 알림을 보낸다.
- Slack 매칭이 실패하면 다우 알림 API 사용을 검토한다.
- 예약 해제는 다우오피스의 공식 취소 API 또는 승인된 자동화 경로가 있을 때만 자동 처리한다.
- 공식 경로가 없으면 취소 링크 또는 취소 화면 경로 안내로 대체한다.

## 8. 주요 데이터

제품이 신뢰성 있게 동작하려면 회의실 메타데이터가 필요하다.

다우오피스 원본에서 가져온다고 가정하는 정보:

- 회의실 이름
- 층
- 호수
- 수용 인원
- 예약 가능 시간 정보
- 예약자 이름
- 회의실 자산 정보

Slack 연동으로 보강하는 정보:

- Slack 사용자 ID
- Slack 표시 이름
- Slack 프로필 링크
- DM 가능한 대상 여부

개인 일정 조회에 필요한 정보:

- 사용자 식별자
- 사용자 캘린더 이벤트
- 이벤트와 회의실 연결 정보

## 9. 응답 원칙

- 조회 결과는 단순 목록보다 추천 중심으로 보여준다.
- 같은 조건이면 적정 규모의 회의실을 우선 추천한다.
- 예약자 식별이 애매하면 잘못된 사람을 확정하지 않는다.
- "사용 중"과 "예약 중"은 구분 가능한 경우만 구분하고, 아니면 예약 기준이라고 명시한다.

## 10. 성공 기준

출시 후 아래 변화를 기대한다.

- 직원이 회의실을 찾는 데 걸리는 시간이 줄어든다.
- 회의실 위치를 찾느라 헤매는 일이 줄어든다.
- 인원수에 맞지 않는 과대 회의실 사용이 줄어든다.
- 예약자 확인과 연락 시간이 줄어든다.
- 미사용 예약 정리율이 높아진다.

## 11. 가정 및 제약

- 회의실 메타데이터(층/호수/수용인원)는 다우오피스 원본에 존재한다고 가정한다.
- 내 다음 회의 조회는 다우오피스 캘린더 등록이 되어 있는 사용자만 가능하다.
- v1은 실제 물리 점유 여부를 감지하지 않는다.
- 예약 해제 자동화는 다우오피스 측 API 또는 내부 승인된 자동화 경로가 있을 때만 지원한다.
- Slack 매칭은 이메일 기준을 기본으로 하되, 불가능하면 이름 기반 보조 매칭을 사용한다.
- 공개 문서 기준으로 다우오피스 Open API에서 회의실 예약 조회/취소 API는 별도 확인이 필요하다.
