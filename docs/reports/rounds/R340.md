**Scanner가 지적한 실질적 위험**:

1. `MicrometerAgentMetrics.trackMissingQuery`(line 486-495):
   ```kotlin
   aggregate.count.incrementAndGet()       // atomic (1)
   aggregate.lastOccurredAt = Instant.now() // volatile write (2)
   ```
2. `@Volatile` 선언이 있어 write 가시성은 보장되지만 **"두 write 사이 순서 보장"**은 아니다. 병렬 스레드가 동일 cluster로 호출할 때:
   - Thread A: (1a) count=5 → (2a) T1 획득 → ... 지연
   - Thread B: (1b) count=6 → (2b) T2=now 획득 → write lastOccurredAt=T2
   - Thread A: write lastOccurredAt=T1 (지연 write이므로 T2를 덮어씀, T1 < T2)
3. 결과: `topMissingQueries()`가 `MissingQueryInsight(count=6, lastOccurredAt=T1)`을 반환. count는 최신이지만 timestamp는 stale.

#### Fix

**1. `MissingQueryAggregate` 필드 변경** (`ResponseValueInsights.kt`):

```kotlin
// BEFORE
@Volatile var lastOccurredAt: Instant = Instant.now()

// AFTER
val lastOccurredAt: AtomicReference<Instant> = AtomicReference(Instant.now())
```

`var` → `val` (AtomicReference는 immutable reference, 내부 값만 atomic update). `import java.util.concurrent.atomic.AtomicReference` 추가.

**2. `trackMissingQuery` update 경로** (`MicrometerAgentMetrics` + `AgentMetrics` 두 곳):

```kotlin
aggregate.count.incrementAndGet()
val now = Instant.now()
aggregate.lastOccurredAt.updateAndGet { prev ->
    if (prev == null || now.isAfter(prev)) now else prev
}
```

`updateAndGet`의 closure는 CAS(compare-and-swap) 루프로 "현재 값보다 더 미래면 교체, 아니면 유지"를 atomic하게 수행. 어떤 스레드가 먼저 쓰든 결과는 "최댓값"으로 수렴하며 JMM race가 불가능해진다.

**3. `topMissingQueries` read 경로** (두 구현체):

```kotlin
.thenByDescending { it.lastOccurredAt.get() }  // 정렬
// ...
lastOccurredAt = it.lastOccurredAt.get(),  // MissingQueryInsight 생성
```

`MissingQueryInsight.lastOccurredAt: Instant`는 public 데이터 클래스라 변경하지 않고, 내부 AtomicReference에서 `.get()`으로 현재 값 스냅샷을 꺼낸다.

#### 설계 메모

- **대안 1**: `synchronized(aggregate) { count++; lastOccurredAt = now }` 블록. 채택 안 함 — synchronized는 monitor contention을 유발하며 `count`는 이미 AtomicLong이라 lock 없이 atomic하게 동작해야 효율적. `AtomicReference`가 lock-free.
- **대안 2**: `count`와 `lastOccurredAt`을 하나의 record로 합쳐 `AtomicReference<Pair<Long, Instant>>`로 관리. 채택 안 함 — pair 교체는 `updateAndGet`에서 매 호출마다 Pair 객체를 allocate하므로 GC 압박 증가. 두 atomic primitive로 분리하는 것이 더 효율적.
- **대안 3**: lastOccurredAt은 그대로 두고 count 후에 `Thread.onSpinWait()`으로 순서 유지 시도. 채택 안 함 — 작동 보장 없음, hack.
- **채택**: `AtomicReference<Instant>` + `updateAndGet(max)` 패턴. 교과서적 lock-free 동시성 해결책.
