**두 코드 경로의 의도 불일치 (실제 증거)**:

1. `VerifiedSourcesResponseFilter.kt:354`:

   → `spec_`이 **workspace(엔터프라이즈 내부) 도구**로 명시적 인식.

2. `ExecutionResultFinalizer.kt` 원본 `toolFamily`:
   ```kotlin
   private fun toolFamily(toolName: String): String {
       return when {
           toolName.startsWith("confluence_") -> "confluence"
           toolName.startsWith("jira_") -> "jira"
           toolName.startsWith("bitbucket_") -> "bitbucket"
           toolName.startsWith("work_") -> "work"

   ```
   → `spec_` 분기 없음. `spec_detail`, `spec_search` 같은 Swagger/OpenAPI 도구 호출이 `"other"` bucket에 묻힘.

**영향 경로**:
- `finalize` → `recordResponseObservationEvent` → `responseObservationMetadata` → `metadata["toolFamily"] = deriveToolFamily(toolsUsed)` → `agentMetrics.recordResponseObservation(event)` → `MicrometerAgentMetrics.incrementBucket(toolFamilyCounts, toolFamily, "none")` → Caffeine cache의 `"other"` bucket 카운터 증가
- Grafana "tool family usage over time" 패널에서 spec 도구 사용이 `"other"`에 병합되어 구분 불가
- `employee-value-rollout-300.md`의 "tool family correctness" KPI가 spec 도구 사용 분포를 저평가

**실제 Swagger/OpenAPI 도구 예시**:
- `spec_search`: API 스펙 검색
- `spec_detail`: 특정 endpoint 상세 정보
- `spec_list_endpoints`: 엔드포인트 목록

이 도구들은 개발자 직원이 API 레퍼런스를 물을 때 빈번히 호출되는 주요 workspace 도구이므로, metric drift는 제품 가치 분석에 직접 영향.

#### Fix

`toolFamily` 함수에 `spec_` 분기 한 줄 추가:

```kotlin
private fun toolFamily(toolName: String): String {
    return when {
        toolName.startsWith("confluence_") -> "confluence"
        toolName.startsWith("jira_") -> "jira"
        toolName.startsWith("bitbucket_") -> "bitbucket"
        toolName.startsWith("work_") -> "work"
        toolName.startsWith("spec_") -> "spec"   // R344 fix
        toolName.startsWith("mcp_") -> "mcp"
        else -> "other"
    }
}
```

`mcp_` 앞에 배치해 `spec_` 특정 prefix를 먼저 매치한 뒤 일반 MCP로 넘어가도록 함(현재로는 두 prefix가 겹치지 않지만 순서를 의도적으로 배치해 향후 `mcp_spec_` 같은 복합 prefix가 생겨도 규칙 예측 가능).

#### 설계 메모

- **대안 1**: `WORKSPACE_TOOL_PREFIXES`를 공유 상수로 추출하고 `toolFamily`가 이를 기반으로 동적 매핑. 채택 안 함 — 현재 prefix → family 매핑이 1:1 단순하고 도메인별 이름이 다르므로(`confluence_` → `"confluence"`, `spec_` → `"spec"`) 하드코딩이 더 명확. 공유 상수는 "어떤 접두사를 workspace로 간주하는가"와 "그 접두사가 어떤 family 이름에 매핑되는가"가 서로 다른 concern.
- **대안 2**: 완전히 enum화. 채택 안 함 — breaking change, 운영 관측 레이어가 문자열 기반.
- **채택**: `when` 분기 1줄 추가 + KDoc으로 의도 명시. 최소 변경.
- §0.1 MCP 계약: MCP 도구의 wire protocol과 무관. Arc Reactor 내부 분류 레이어만 변경.

- §0.3 대화 컨텍스트: 완전 무관.
