#### 작업 배경

R330 scanner가 MCP 연결 계층에서 LOW 4건을 추가 관찰했다. 그중 #4 `loadToolCallbacks()` 0-tool 허용 → 재연결 루프 유발이 가장 user-visible impact가 큰 항목이었다. R330에서 HIGH 2건(SHARED builder race, handleConnectionError identity)을 처리한 뒤 R331은 LOW 4건 중 이 하나를 직접 가치 기준으로 승격 처리한다.

**실제 코드 흐름 확인**:
1. `McpConnectionSupport.loadToolCallbacks()`는 `client.listTools().tools()` 결과가 빈 경우 `emptyList()`를 반환한다 (McpConnectionSupport.kt:400-409).
2. `DefaultMcpManager.connect()`는 handle이 non-null이면 tools 크기와 무관하게 `toolCallbacksCache.put(name, handle.tools)` + 상태를 CONNECTED로 전이한다 (McpManager.kt:335-338).
3. `McpHealthPinger.checkConnectedHealth()`는 `mcpManager.getToolCallbacks(name).isEmpty()`면 즉시 `attemptReconnectWithCooldown` 호출한다 (R331 fix 전 McpHealthPinger.kt:191).
4. 쿨다운 5분 후 `ensureConnected → connect()` 가 재실행되어 같은 결과(0 tools)를 얻고 CONNECTED로 다시 전이. → step 3 반복 → **무한 루프**.

MCP 프로토콜 상 0-tool 서버는 유효한 상태다. 예: 아직 도구 등록 전인 신규 서버, resource/prompt 전용 서버, 권한 정책으로 도구가 모두 필터링된 상태. 이 경우를 재연결 루프에 태우면 불필요한 TCP 소켓·프로세스 리소스 소모 + 로그 노이즈 + 5분마다 관측성 지표에 불안정 signal을 주입한다.

#### Fix — `seenNonEmptyServers` 기반 degradation 감지

**핵심 아이디어**: 재연결은 "퇴화(degradation) 감지"로만 사용한다. "이전에 non-empty였던 적이 있는 서버가 지금 empty"일 때만 연결이 조용히 끊겼다고 판단하고, 한 번도 non-empty를 본 적 없는 서버는 "안정적 0-tool 서버"로 간주하여 재연결하지 않는다.

**구현** (`McpHealthPinger.kt`):

```kotlin
private val seenNonEmptyServers: Cache<String, Boolean> =
    Caffeine.newBuilder().maximumSize(MAX_TRACKED_SERVERS).build()

private suspend fun checkConnectedHealth(serverName: String) {
    try {
        val tools = mcpManager.getToolCallbacks(serverName)
        if (tools.isNotEmpty()) {
            // 도구가 있으면 baseline 마킹 후 종료 — 이후 empty 전이 시 degradation 감지
            seenNonEmptyServers.put(serverName, true)
            return
        }
        // R331: empty는 "이전에 non-empty였을 때만" degradation
        if (seenNonEmptyServers.getIfPresent(serverName) == true) {
            logger.warn { "MCP 서버 '$serverName'가 이전에 도구가 있었으나 현재 비어있음 — 재연결 시도" }
            seenNonEmptyServers.invalidate(serverName)  // baseline 리셋
            attemptReconnectWithCooldown(serverName)
        } else {
            logger.debug { "MCP 서버 '$serverName' CONNECTED & 도구 0개 — 0-tool 서버로 간주, 재연결 생략 (R331)" }
        }
    } catch (e: Exception) {
        e.throwIfCancellation()
        logger.warn(e) { "MCP 서버 '$serverName' 헬스체크 중 오류 — 재연결 시도" }
        attemptReconnectWithCooldown(serverName)
    }
}
```

**왜 baseline 리셋인가**: `non-empty → empty` 퇴화 감지로 `attemptReconnectWithCooldown`이 실행되면 `seenNonEmptyServers.invalidate`를 호출해 baseline을 리셋한다. 이는 재연결 후 새 상태를 다시 평가하기 위함이다.
- 재연결 성공 + 도구 복구: 다음 주기에서 `tools.isNotEmpty()` → baseline 재마킹
- 재연결 성공 + 여전히 0-tool: 이제 서버가 영구적 0-tool 상태로 판단 → 더 이상 재연결 안 함 ✅
- 재연결 실패(FAILED로 전환): `pingAllConnectedServers` loop에서 FAILED 분기가 별도 쿨다운 재연결 처리

**왜 `seenNonEmptyServers`를 McpHealthPinger에 두었는가**: McpManager 인터페이스에 상태를 추가하면 여러 구현체를 수정해야 하고 테스트 표면이 넓어진다. HealthPinger는 이미 `lastReconnectAttempt` 같은 서버별 상태를 보유하므로 도메인 상 여기 두는 것이 자연스럽다. Caffeine bounded cache로 unbounded growth도 방지.

- 대안 2: `checkConnectedHealth`에서 `client.listTools()`로 active probe. 연결 생존을 직접 확인할 수 있으나 `McpConnectionSupport.loadToolCallbacks`와 구조가 중복되고 비용/복잡성 증가.
- 채택: HealthPinger 내부 트래커. 최소 변경으로 현재 문제를 해결하고 이전 behavior(퇴화 감지)를 보존.
