#### Fix — side-channel metadata 접근

기존 `checkToolApproval` 체인의 `String?` signature를 그대로 유지하고 side-channel metadata 키로 modified args를 실행 경로에 전달:

**1. 상수 추가** (`ToolCallOrchestrator.companion object`):

```kotlin
const val HITL_MODIFIED_ARGS_KEY_PREFIX = "hitlModifiedArgs_"
```

**2. `recordApprovalMetadata` — 승인 시 저장** (기존 `hitlWaitMs_`, `hitlApproved_` 패턴 재사용):

```kotlin
if (response.approved) {
    // ...
    response.modifiedArguments?.let { modified ->
        if (modified.isNotEmpty()) {
            hookContext.metadata[HITL_MODIFIED_ARGS_KEY_PREFIX + keySuffix] = modified
            logger.info { "... HITL 승인 시 파라미터 수정 요청 기록 ..." }
        }
    }
    return null
}
```

**3. `applyApprovedModifications` 헬퍼** — 실행 경로에서 읽어 `ToolCallContext` copy:

```kotlin
private fun applyApprovedModifications(
    toolCallContext: ToolCallContext,
    hookContext: HookContext
): ToolCallContext {
    val key = HITL_MODIFIED_ARGS_KEY_PREFIX + hitlMetadataSuffix(toolCallContext)
    @Suppress("UNCHECKED_CAST")
    val modified = hookContext.metadata[key] as? Map<String, Any?> ?: return toolCallContext
    logger.info { "R336: HITL 승인 시 수정된 파라미터를 실행 경로에 적용 — ..." }
    return toolCallContext.copy(toolParams = modified)
}
```

**4. 단일 경로** (`executeDirectToolCall`):

```kotlin
checkToolApproval(toolName, toolCallContext, hookContext)?.let { rejection -> ... }

// R336
val finalizedContext = applyApprovedModifications(toolCallContext, hookContext)
val finalizedParams = finalizedContext.toolParams

return invokeAndFinalizeDirect(
    effectiveToolParams = finalizedParams,
    toolCallContext = finalizedContext,
    ...
)
```

**5. 병렬 경로** (`executeSingleToolCall`) — `toolInput`(serialized JSON)도 재직렬화:

```kotlin
checkHookAndApproval(...)?.let { return it }

// R336
val finalizedContext = applyApprovedModifications(prepared.context, hookContext)
val finalizedToolInput = if (finalizedContext !== prepared.context) {
    serializeToolInput(finalizedContext.toolParams, toolCall.arguments())
} else {
    prepared.toolInput
}

return invokeAndFinalizeParallel(
    toolInput = finalizedToolInput,
    toolCallContext = finalizedContext,
    ...
)
```

#### 설계 메모

- **대안 1**: `checkToolApproval` 반환 타입을 `ApprovalOutcome` sealed class(Approved/Rejected)로 변경. 채택 안 함 — 4~5개 private method signature 변경 + 두 경로 모두 when 분기 수정으로 blast radius가 크고, 새벽 자동 실행 모드의 "작은 scope" 원칙에 부합 안 함. side-channel 접근이 `hitlWaitMs_`/`hitlApproved_`/`hitlRejectionReason_` 기존 key 패턴과 일관성 있고 최소 변경이다.
- **대안 2**: `ToolCallContext`를 mutable로 변경해 in-place 수정. 채택 안 함 — `data class val toolParams` 변경은 여러 소비자에게 영향을 주고 불변성 규약을 깨뜨림.
- **대안 3**: `ToolApprovalResponse`를 execute 경로까지 전달. 채택 안 함 — approval 전용 타입을 실행 시그니처에 오염시킴.
- **채택**: side-channel metadata. `hitlMetadataSuffix(toolCallContext)`가 이미 per-call 고유 키를 생성하므로 병렬 tool call에서도 충돌 없음.
- §0.1 MCP 계약 불변: MCP wire protocol / 도구 발견 / tool description 모두 무관. modified arguments는 Arc Reactor 내부 경로에만 영향.
- §0.2 Redis cache: `systemPrompt`/`scopeFingerprint` 불변. metadata 키만 추가.
- §0.3 대화 컨텍스트: 완전 무관.
