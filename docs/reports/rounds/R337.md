#### 작업 배경

R336이 `safe_action_workflows` axis 전환과 함께 P1 HIGH(HITL modifiedArguments silent 무시)를 처리했고, R337은 동일 scanner batch의 LOW #3(Caffeine evict silent 승인 실패)를 같은 axis에서 이어서 처리한다. 작은 scope이지만 user-visible UX 버그이므로 direct_value.

**버그의 실체**:

1. `InMemoryPendingApprovalStore`는 R310에서 `ConcurrentHashMap → Caffeine bounded cache`로 마이그레이션됨. `maximumSize = 10_000` (기본).
2. 대량 승인 요청이 몰리면 Caffeine이 W-TinyLFU 정책으로 오래된/프리퀀시 낮은 entry를 evict.
3. evict는 단순히 cache에서 제거하는 동작이고, 제거된 `PendingEntry`의 `CompletableDeferred`는 **완료되지 않는다**.
4. 결과 1 (사용자 측): `requestApproval`의 `withTimeoutOrNull(effectiveTimeout) { deferred.await() }`가 전체 effectiveTimeout(기본 5분)까지 기다렸다가 타임아웃.
5. 결과 2 (관리자 측): REST API로 `approve(approvalId)` 호출 → `pending.getIfPresent(approvalId) == null` → `false` 반환 → 관리자 UI에 "승인 실패"로 표시.
6. 관리자와 사용자 양쪽 모두 silent하게 잘못된 상태를 경험.

기존 R310 주석은 "evict된 엔트리는 `CompletableDeferred`가 완료되지 못하고 `withTimeoutOrNull`에서 자연스럽게 타임아웃되므로 coroutine leak 없음"이라고 설명하고 있으나, **타임아웃까지의 장시간 대기 + 관리자 approve 실패**라는 두 가지 UX 결함은 "coroutine leak 없음"의 범위 밖이었다.

#### Fix

Caffeine builder에 `removalListener`를 추가해 `SIZE` cause evict 시 대기 중인 `deferred`를 overflow 응답으로 즉시 완료:

```kotlin
private val pending: Cache<String, PendingEntry> = Caffeine.newBuilder()
    .maximumSize(maxPending)
    .removalListener<String, PendingEntry> { key, entry, cause ->
        if (cause == RemovalCause.SIZE && entry != null) {
            val completed = entry.deferred.complete(
                ToolApprovalResponse(
                    approved = false,
                    reason = "Approval store overflow — request dropped (maxPending exceeded)"
                )
            )
            if (completed) {
                logger.warn {
                    "R337: 승인 저장소 overflow evict 로 대기 엔트리 드롭 — " +
                        "id=$key, tool=${entry.request.toolName}, user=${entry.request.userId}"
                }
            }
        }
    }
    .build()
```

**`SIZE`만 처리하는 이유**:

| RemovalCause | 발생 시점 | R337 처리 |
|---|---|---|
| `EXPLICIT` | `invalidate(id)` 호출 (approve/reject finally 블록) | 무시 — deferred는 이미 `complete`됨 |
| `REPLACED` | `put(id, ...)` 재호출 | 무시 — 동일 키 replace는 실제 운영 흐름에 없음 |
| `EXPIRED` | time-based expiration | 무시 — R337 시점에서는 `expireAfter` 미설정 |
| `COLLECTED` | GC 기반 reference 수집 | 무시 — R337 시점에서는 weak/soft reference 미사용 |
| `SIZE` | `maximumSize` 초과로 evict | **R337 처리** — overflow 응답 전달 |

**이중 완료 안전성**: `approve()`/`reject()` 경로는 이미 `entry.deferred.complete(response)`를 먼저 호출하고 `requestApproval`의 `finally` 블록에서 `pending.invalidate(id)`를 호출한다. `invalidate`는 `EXPLICIT` cause라 removalListener의 `SIZE` 분기에 들어가지 않으므로 이중 완료 없음. 설령 race로 어떤 경로가 두 번 `complete`을 시도해도 `CompletableDeferred.complete`는 idempotent이며 이미 완료된 deferred에 대해 단순히 `false`를 반환한다.

**`forceCleanUp` helper**: Caffeine의 eviction은 **비동기**로 스케줄되므로 `put` 직후 즉시 관찰되지 않는다. 테스트가 R337 동작을 deterministic하게 검증하려면 `pending.cleanUp()`을 명시적으로 호출해 maintenance 태스크를 drain해야 한다. production 경로는 이 helper를 호출하지 않고 Caffeine의 자동 maintenance에 맡긴다.

#### 설계 메모

- **대안 1**: `deferred.completeExceptionally(RuntimeException("overflow"))`. 채택 안 함 — 호출자 `requestApproval`이 일반 응답으로 처리하게 하는 것이 호환성이 높다(기존 catch 블록을 수정할 필요 없음).
- **대안 2**: `expireAfterWrite` 추가로 stale entry 자동 만료. 채택 안 함 — 기본 `defaultTimeoutMs`가 이미 coroutine 레벨 타임아웃을 제공하므로 이중 타임아웃은 중복.
- **대안 3**: `RemovalCause.wasEvicted()`로 모든 eviction 원인 처리. 채택 안 함 — `EXPIRED`/`COLLECTED`는 R337 시점에 사용되지 않지만 향후 `expireAfter` 추가 시 `wasEvicted()`에 포함되어 예상치 못한 동작을 유발할 수 있다. `SIZE` 명시적 분기가 더 안전.
- **채택**: `SIZE` 분기 + `CompletableDeferred.complete` + log + forceCleanUp 테스트 helper.
