#### 작업 배경

R336 scanner가 `safe_action_workflows` axis 전환 시 발견한 LOW 5건 중 #1을 R338에서 처리한다. R336에서 P1 HIGH(modifiedArguments silent 무시), R337에서 LOW #3(InMemoryPendingApprovalStore evict silent 실패)을 처리했고, R338은 JDBC 저장소 측의 동등 수준 결함을 마무리한다.

**버그의 실체 — DB 영구 누적 경로**:

1. `JdbcPendingApprovalStore.requestApproval` 폴링 루프(line 78~108)는 매 `pollIntervalMs`(기본 250ms)마다 `findState(id)`를 호출해 상태 변화를 감지한다.
2. 폴링 중 `ApprovalStatus.TIMED_OUT`이 감지되면 (line 101~106) 즉시 early return한다. 이 경로에서는 **단계 3 UPDATE(line 111~124)가 실행되지 않는다** — 단계 3 UPDATE는 while 루프가 deadline 도달로 종료된 경우에만 실행되고, early return은 while 밖으로 빠져나온 것이므로 건너뛴다.
3. 누가 먼저 `TIMED_OUT`을 만들었는가? 가능성:
   - **다중 인스턴스 race**: 인스턴스 A가 `requestApproval` → 행 insert → 폴링 중 → 어떤 이유로 다른 프로세스가 수동/자동 스크립트로 행을 `TIMED_OUT`으로 마킹(예: DB 정리 job이 stale PENDING을 TIMED_OUT으로 승격시키되 resolved_at 세팅 누락)
   - **legacy migration**: 이전 버전이 해결 상태를 만들면서 `resolved_at` 세팅을 누락
   - **외부 DBA 조작**: 테스트/디버깅을 위해 수동 SQL로 행 상태 변경
   - **버그가 있는 스크립트**: 배치 job이 `UPDATE ... SET status = 'TIMED_OUT'` 쿼리를 `resolved_at` 없이 실행
4. 결과: 행이 `status = TIMED_OUT`, `resolved_at = NULL`, `requested_at = <n일 전>` 상태로 남는다.
5. `cleanupResolvedRows` DELETE 쿼리는 `AND resolved_at IS NOT NULL AND resolved_at < ?`이므로 이런 행에 **절대 걸리지 않는다**.
6. retention 기간(기본 7일)과 관계없이 해당 행이 DB에 **영구 잔류**한다. 관리/관측 쿼리가 "오래된 타임아웃 행"을 계속 보게 된다.

실질적 impact는 조건부지만, 장기적 silent growth에 대한 방어가 필요하다. Scanner가 LOW로 분류한 것은 "자연 발생 시나리오가 명확하지 않음" 때문이고, 실제 근본 해결 가치는 MED 수준.

#### Fix — cleanup 쿼리 OR fallback

`cleanupResolvedRows`의 DELETE WHERE 절을 확장:

```sql
DELETE FROM pending_approvals
 WHERE status IN (?, ?, ?)
   AND (
       (resolved_at IS NOT NULL AND resolved_at < ?)
       OR (resolved_at IS NULL AND requested_at < ?)
   )
```

파라미터 바인딩:
- `status`: APPROVED, REJECTED, TIMED_OUT (기존 동일)
- 첫 cutoff: `resolved_at < ?` (정상 경로)
- 두 번째 cutoff: `requested_at < ?` (fallback) — 같은 `Timestamp.from(cutoff)` 객체를 재사용

**의미**: `resolved_at`이 세팅된 정상 경로는 retention 윈도우를 `resolved_at` 기준으로 측정(기존과 동일). `resolved_at`이 NULL인 병적 행은 `requested_at` 기준으로 측정(fallback). 두 경로 모두 동일한 cutoff 시점을 참조한다.

#### 설계 메모

- **대안 1**: 폴링 루프의 `TIMED_OUT` 분기에 defensive UPDATE (`SET resolved_at = NOW() WHERE ... AND resolved_at IS NULL`). 채택 안 함 — 이는 "폴링 경로에서만" 병적 상태를 복구하므로 외부 원인(DBA, 스크립트, migration)은 여전히 커버 못함. cleanup 쪽에서 근본 해결하는 것이 포괄적.
- **대안 2**: `status = TIMED_OUT`만 fallback 적용. 채택 안 함 — `APPROVED`/`REJECTED`도 외부 원인으로 NULL이 될 수 있으므로 모든 resolved 상태에 동일하게 적용하는 것이 일관성 있다.
- **대안 3**: `requested_at + timeout_ms < NOW()`로 더 정교하게. 채택 안 함 — retention은 "resolved 이후 보관 기간"이고 timeout은 "승인 대기 허용 시간"이라 의미가 다르다. 단순히 `requested_at < cutoff`가 기존 retention 의미와 더 일치.
- **채택**: OR fallback with cutoff 재사용. 최소 변경 + 근본 해결.
