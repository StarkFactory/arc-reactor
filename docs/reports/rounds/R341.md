#### 작업 배경

R339~R340에서 `admin_productization` scanner defer 중 P2 MED #1 (`reason` tag cardinality normalize)과 P2 MED #3 (`lastOccurredAt` race)을 처리했고, R341은 마지막 P2 MED #2 (`channel` tag unbounded)를 처리해 **R339 scanner batch를 완결**한다.

**실제 경로 분석**:

1. `recordUnverifiedResponse(metadata)` (line 281):
   ```kotlin
   Counter.builder(METRIC_UNVERIFIED_RESPONSES)
       .tag("channel", metadata["channel"]?.toString()?.ifBlank { "unknown" } ?: "unknown")
       .register(registry)
       .increment()
   ```
   → `channel`이 `"C012ABC34DE"` 같은 Slack 채널 ID일 때 Prometheus에 `arc_agent_responses_unverified_total{channel="C012ABC34DE"}` 시계열이 생성된다. 새 채널이 등록될 때마다 1개씩 추가되어 무제한 성장.

2. `recordStageLatency(stage, durationMs, metadata)` (line 425):
   ```kotlin
   Timer.builder(METRIC_STAGE_DURATION)
       .tag("stage", stage)
       .tag("channel", metadata["channel"]?.toString()?.trim()?.ifBlank { "unknown" } ?: "unknown")
       ...
   ```
   → 동일 패턴. Timer도 tag별 시계열을 생성하므로 `{stage, channel}` 조합이 카디널리티 제곱적 폭발 위험.

**상대적으로 `channelCounts` Caffeine cache는 보호됨**:
- `recordResponseObservation` → `incrementBucket(channelCounts, ...)`: Caffeine `maximumSize=10_000`으로 eviction
- 이 bucket은 `ResponseValueSummary.channelCounts` API 응답에만 사용, Micrometer 레지스트리와 독립

**두 경로의 불일치**: Caffeine으로 보호되는 경로 vs Micrometer 직접 등록 경로가 공존하고 후자는 정책이 없었음. R341은 후자에 동일한 bounded 정책을 적용.

#### Fix

**1. Budget cache 선언** (`MicrometerAgentMetrics.kt`):

```kotlin
private val unverifiedChannelTagBudget: Cache<String, Boolean> =
    Caffeine.newBuilder().maximumSize(MAX_UNVERIFIED_CHANNEL_TAGS).build()
```

**2. 공유 helper `boundedChannelTag`**:

```kotlin
private fun boundedChannelTag(rawChannel: String?): String {
    val trimmed = rawChannel?.trim()?.takeIf { it.isNotBlank() } ?: return UNKNOWN_CHANNEL_TAG
    val cleaned = trimmed.take(MAX_CHANNEL_TAG_LENGTH)
    if (unverifiedChannelTagBudget.getIfPresent(cleaned) != null) return cleaned
    if (unverifiedChannelTagBudget.asMap().size < MAX_UNVERIFIED_CHANNEL_TAGS) {
        unverifiedChannelTagBudget.put(cleaned, true)
        return cleaned
    }
    return OVERFLOW_CHANNEL_TAG
}
```

**흐름**:
1. null/blank → `"unknown"`
2. 길이 상한 `MAX_CHANNEL_TAG_LENGTH=64`자 절단
3. budget cache에 이미 있으면 재사용 (LRU 재방문 hit)
4. budget size < 상한이면 추가 후 그대로 사용
5. 상한 초과면 `"other"`로 폴백

**3. Companion object 상수 추가**:

```kotlin
internal const val MAX_UNVERIFIED_CHANNEL_TAGS = 128L
internal const val MAX_CHANNEL_TAG_LENGTH = 64
internal const val UNKNOWN_CHANNEL_TAG = "unknown"
internal const val OVERFLOW_CHANNEL_TAG = "other"
```

**4. 두 call site에서 helper 사용**:

```kotlin
// recordUnverifiedResponse
val channelTag = boundedChannelTag(metadata["channel"]?.toString())
Counter.builder(METRIC_UNVERIFIED_RESPONSES)
    .tag("channel", channelTag) // R341
    .register(registry).increment()

// recordStageLatency
Timer.builder(METRIC_STAGE_DURATION)
    .tag("stage", stage)
    .tag("channel", boundedChannelTag(metadata["channel"]?.toString())) // R341
    .register(registry).record(...)
```

**5. RecentTrustEvent는 건드리지 않음**:
- `MAX_TRUST_EVENTS=100`으로 이미 in-memory deque가 bounded
- dashboard drill-down용 raw 값 보존이 필요하므로 그대로 유지
- Micrometer 레지스트리 카디널리티와 별개 경로

- **대안 1**: `channelCounts` Caffeine cache를 공유 source of truth로 사용. 채택 안 함 — bucket 업데이트는 `recordResponseObservation`만 호출하는데, `recordUnverifiedResponse` / `recordStageLatency`는 독립 경로라 순서 의존성 발생. bucket이 비어있는 상태에서 unverified가 먼저 호출되면 "unknown"으로 폴백되어 정보 손실.
- **대안 2**: `channel` 태그 자체를 제거. 채택 안 함 — 운영자가 채널별 집계를 필요로 하고 R282에서 명시적으로 bucket으로 유지했다.
