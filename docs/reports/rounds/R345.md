**버그의 실체**:

기존 구현:
```kotlin
private fun stripSourcesBlock(content: String): String {
    val trimmed = content.trimEnd()
    val matches = SOURCE_HEADING_REGEX.findAll(trimmed).toList()
    if (matches.isEmpty()) return trimmed
    val match = matches.first()
    return trimmed.substring(0, match.range.first).trimEnd()
}
```

`SOURCE_HEADING_REGEX = Regex("(?m)^\\s*(?:\\*\\*)?(Sources|출처)(?:\\*\\*)?(?:\\s*:\\s*.*)?$")`

즉 `Sources`, `Sources: ...`, `**Sources**`, `**Sources:** ...`, `출처`, `출처: ...`, `**출처:**` 등을 모두 line 단위로 매칭. 첫 번째 매칭 위치부터 본문 끝까지 **무조건 잘라낸다**.

**시나리오 — user-visible 절단 버그**:

```
The approach I used was:

Sources:
- the internal knowledge base (not linked here)
- previous team discussions

Given the above, my conclusion is that we should ship feature X next week.
```

LLM이 이런 응답을 작성한 경우:
- `matches.first()` = `"Sources:"` 단독 라인
- `substring(0, Sources: 위치).trimEnd()` = `"The approach I used was:"`
- **전체 결론 "Given the above, my conclusion is that we should ship feature X next week." 가 사라진다**

직원은 "The approach I used was:" 뒤에 공백만 남은 불완전한 답변을 받는다. `employee_value` axis의 핵심 지표(답변 완전성/신뢰도)에 직접 영향.

**기존 테스트가 lock하는 정상 케이스**:

Arc Reactor가 프롬프트 레이어에서 LLM에게 지시하는 "Sources 블록 형식"은 항상 bullet + markdown link. 기존 테스트 3건이 이 정상 케이스의 제거를 lock:

1. `appending normalized footer 전에 strip leaked tool code and generated sources block` — `Sources:\n* Jira project BACKEND: https://...` (bullet + URL)
2. `strip inline and emphasized sources sections from tool output` — `Sources: https://.../weekly\n\n**출처:**\n* Jira project DEV: https://...` (multiple forms)

따라서 fix는 **정상 Sources 블록은 제거**하면서 **서술적 언급은 보존**하는 이중 조건이 필요.

#### Fix — 이중 조건: 헤딩 매칭 + 아래 내용 검증

```kotlin
private fun stripSourcesBlock(content: String): String {
    val trimmed = content.trimEnd()
    val matches = SOURCE_HEADING_REGEX.findAll(trimmed).toList()
    if (matches.isEmpty()) return trimmed
    val first = matches.first()
    val below = trimmed.substring(first.range.last + 1)
    if (below.isBlank()) {
        return trimmed.substring(0, first.range.first).trimEnd()
    }
    val hasSourceListLine = below.lineSequence().any { line ->
        val t = line.trim()
        if (t.isEmpty()) return@any false
        val firstChar = t.firstOrNull()
        (firstChar in BULLET_CHARS && (t.contains("http") || t.contains("]("))) ||
            t.startsWith("http")
    }
    return if (hasSourceListLine) {
        trimmed.substring(0, first.range.first).trimEnd()
    } else {
        trimmed
    }
}
```

**정책 논리**:
1. Sources 헤딩 매칭이 없으면 본문 그대로 반환
2. 헤딩 뒤에 아무 내용 없으면 (`below.isBlank()`) 단순 제거
3. 헤딩 뒤 **어떤 라인이라도** 다음 패턴을 만족하면 sources block으로 판단 → 첫 매칭부터 절단:
   - bullet(`-`, `*`)로 시작하고 `http` 또는 `](` 포함 (markdown link bullet)
   - 또는 `http`로 시작 (bare URL)
4. 위 패턴이 전혀 없으면 서술적 언급 가능성 → 본문 보존 (false negative 안전)

**Companion object 상수 추가**:
```kotlin
private val BULLET_CHARS = setOf('-', '*')
```

#### 시나리오별 동작 검증

| 시나리오 | 기존 동작 | 수정 후 동작 |
|---|---|---|
| `Sources\n- [title](url)\n- [t2](url2)` (정상 Arc Reactor 출력) | 제거 ✓ | 제거 ✓ (bullet+`](` 감지) |
| `Sources:\n* Jira project: https://...` (LLM 생성) | 제거 ✓ | 제거 ✓ (bullet+`http` 감지) |
| `Sources: https://single.example.com` (inline URL) | 제거 ✓ | 제거 ✓ (다른 line에 `http`, 또는 정규식 매칭 범위가 URL 포함) |
| `**출처:**\n* Jira project DEV: https://...` | 제거 ✓ | 제거 ✓ (bullet+http) |
| `Sources:\n- ref 1\n- ref 2\n\n결론 X` (서술적, URL 없음) | **본문 절단** ✗ | **본문 보존** ✓ (R345 fix) |

| `Sources` (헤딩만 있고 뒤 없음) | 제거 ✓ | 제거 ✓ (below.isBlank) |

#### 설계 메모

- **대안 1**: `matches.last()`로 맨 끝 Sources 헤딩만 검사. 채택 안 함 — 기존 테스트 `strip inline and emphasized sources sections`는 "앞에 있는 `Sources: https://...`도 제거돼야 한다"고 lock하고 있어 `matches.last()`로 검사하면 앞 매칭을 놓침. `matches.first()` + 이후 content check 조합이 backward compatible.
- **대안 2**: 첫 non-blank 라인만 검사. 채택 안 함 — 서술적 케이스에서 첫 라인이 URL이 아닌 bullet 텍스트(`- ref 1`)일 때 정확 판단 불가. `below` 전체 라인을 스캔하는 것이 더 robust.
- **대안 3**: LLM에게 "본문 중간에 Sources 쓰지 말라"고 프롬프트 지시. 채택 안 함 — prompt layer 변경은 `scopeFingerprint` 영향 가능(§0.2), 또 LLM이 지시를 100% 따른다는 보장 없음.
- **채택**: 첫 매칭 유지 + below content pattern 검증. 기존 테스트 31건 PASS 유지 + 새 edge case 방어.

- §0.2 Redis cache: `scopeFingerprint` 구성 요소 불변. response filter는 response-time 처리.
- §0.3 대화 컨텍스트: 완전 무관.
