**버그의 root cause**:

1. `MicrometerEvaluationMetricsCollector.recordSafetyRejection(stage, reason)` (line 96-106)이 `reason` 파라미터를 `.tag(TAG_REASON, reason.ifBlank { UNKNOWN_TAG })`로 직접 바인딩.
2. caller `EvaluationMetricsHook.recordSafetyRejections`는 `blockReason ?: errorCode`를 전달.
3. `blockReason`은 `HookContext.metadata["blockReason"]`에 Guard/Hook이 설정하는 **자유 문자열** — `recordSafetyRejection` KDoc이 명시한 소스는 "Guard, OutputGuard, Hook"이지만 value format 계약이 없다.
4. 커스텀 Guard가 "사용자 ID 123에서 PII 감지: email=alice@example.com"처럼 동적 ID를 포함하면 매 호출마다 새로운 tag 시계열이 Micrometer 레지스트리에 등록 → **영구적 카디널리티 성장**.
5. `EvaluationMetricsCatalog`(R234) 문서는 `reason` tag가 enum-like로 소수 bucket에 수렴할 것을 가정하지만 구현이 이 계약을 강제하지 않는다.

**실제 영향**:
- Prometheus scrape 크기가 시계열 수에 선형 → 장기 운영 중 `/actuator/prometheus` 응답 시간 증가
- Time-series DB(Thanos/Cortex/Victoria) 저장 비용 선형 증가
- Grafana 대시보드가 "top safety rejection reasons" 패널을 그릴 때 수천 개 bucket 중 유의미한 소수를 찾기 어려움 → **운영자 가치 저하** (`admin_productization` axis 직접 영향)
- 관련 alerts(`arc.reactor.eval.safety.rejection` rate)가 노이즈에 묻힘

#### Fix

`EvaluationMetricsHook`에 private `normalizeReasonTag(reason: String): String` helper 추가 + `recordSafetyRejections` 호출 직전 normalize:

```kotlin
// BEFORE
val reason = blockReason ?: errorCode ?: MicrometerEvaluationMetricsCollector.UNKNOWN_TAG
collector.recordSafetyRejection(stage, reason)

// AFTER
val rawReason = blockReason ?: errorCode ?: MicrometerEvaluationMetricsCollector.UNKNOWN_TAG
collector.recordSafetyRejection(stage, normalizeReasonTag(rawReason))
```

```kotlin
private fun normalizeReasonTag(reason: String): String {
    if (reason.isBlank()) return MicrometerEvaluationMetricsCollector.UNKNOWN_TAG
    val sanitized = reason.take(MAX_REASON_TAG_LENGTH).replace(REASON_TAG_SANITIZE_REGEX, "_")
    return sanitized.ifBlank { MicrometerEvaluationMetricsCollector.UNKNOWN_TAG }
}

companion object {
    internal const val MAX_REASON_TAG_LENGTH = 64
    private val REASON_TAG_SANITIZE_REGEX = Regex("[^A-Za-z0-9_.\\-]")
}
```

**정규화 정책**:

1. `isBlank()` → `"unknown"` 폴백 (Micrometer 기본 동작 유지)
2. 상위 `MAX_REASON_TAG_LENGTH=64`자만 유지 — 길이 상한으로 긴 쿼리 snippet 차단
3. `A-Za-z0-9_.\-` 외 문자는 모두 `_`로 치환 — 공백·한글·특수문자·@·#·: 등을 정규화
4. 치환 후 결과가 비어있으면 `"unknown"` 폴백 (모든 문자가 치환 대상이었던 극단 케이스)

**왜 caller 위치 normalize인가**:

- Collector 인터페이스(`EvaluationMetricsCollector`)는 여러 구현체(NoOp, MicrometerEvaluationMetricsCollector, 테스트 double)를 지원한다. 모든 구현체를 동시에 수정하는 것은 scope가 크다.
- caller 위치 normalize는 Hook 한 곳에서 모든 collector를 보호 → minimum-invasive.
- `normalizeReasonTag`를 private helper로 유지해 외부 API 변화 없음.
- Scanner가 제안한 접근과 일치.

#### 설계 메모

- **대안 1**: `MicrometerEvaluationMetricsCollector.recordSafetyRejection` 내부에서 normalize. 채택 안 함 — 다른 collector 구현체가 이미 정규화된 값을 받거나 raw 값을 받는 계약 불일치가 발생. Hook에서 한 번 normalize하면 모든 collector에 일관된 값이 전달.
- **대안 2**: `blockReason` 설정 지점(Guard/Hook)에서 normalize 강제. 채택 안 함 — Guard 구현체가 여러 개이고 CustomGuard도 허용하므로 각 구현체마다 정규화 책임을 지울 수 없다.
- **대안 3**: 더 엄격한 허용 문자 집합(소문자만, `_`만). 채택 안 함 — `GUARD_REJECTED` 같은 기존 errorCode가 대문자이므로 대소문자 허용이 기존 의미 보존.

- **채택**: caller 측 normalize + helper 함수 + const/regex in companion. 최소 변경으로 최대 가치.
