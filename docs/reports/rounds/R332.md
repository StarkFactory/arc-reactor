#### 작업 배경

**R330 scan**에서 발견된 LOW 관찰 중 #2(`McpHealthPinger.pingAllConnectedServers`가 PENDING을 처리하지만 `reconnection.enabled=false`일 때 방치되는 edge case)를 직접 조사하다가 더 큰 root cause를 발견했다. `reconnection.enabled` 여부와 무관하게, **두 모듈의 PENDING 처리 의도가 어긋나 있었다**:

| 모듈 | PENDING 처리 |
|---|---|
| `McpHealthPinger.pingAllConnectedServers` (R173) | FAILED/PENDING → `attemptReconnectWithCooldown` 호출 (재연결 시도) |
| `DefaultMcpManager.ensureConnected` | PENDING → `false` 즉시 반환 (재연결 거부) |

`attemptReconnectWithCooldown`의 실체는 `mcpManager.ensureConnected(serverName)`이다. 따라서 R173의 PENDING 경로는 사실상 **no-op**이었고, 기존 테스트 `PENDING 상태의 서버는 자동 재연결을 시도한다`가 mock 기반이어서 이 통합 gap을 잡지 못했다(ensureConnected를 mockk로 stub해 `true`를 반환시키고 호출 여부만 verify).

**`PENDING`이 실제로 쌓이는 시나리오**:

1. `register()` 직접 호출 — `servers.put` + `statuses.put(PENDING)` 후 사용자의 명시적 `connect()`를 기다림 (autoConnect와 무관).
2. `initializeFromStore()` — `servers.put` + `statuses.put(PENDING)` 후 `autoConnect=true`면 즉시 `connect()` 호출. 그러나 예외 발생 시 catch해 로깅만 하고 status는 PENDING에 머무를 수 있다(`connect` 내부는 FAILED로 전이하지만, `connectionSupport.open` 호출 **이전** 실패 경로나 예외 전파 race에서 catch 블록이 먼저 도는 edge case).
3. `syncRuntimeServer()` — 동일 패턴. autoConnect 여부에 따라 후속 경로 상이.

케이스 1은 사용자 수동 제어 의도이므로 health check가 끼어들면 **안 된다**. 케이스 2-3은 "자동 관리 의도였으나 PENDING에 갇힌 상태"이므로 health check가 복구해야 한다.

이 두 케이스를 구분하는 정확한 신호는 **`McpServer.autoConnect` 필드**다. 따라서 R332 fix는 `ensureConnected`에 "PENDING + `autoConnect=true`일 때만 첫 연결 시도"라는 narrow한 분기를 추가한다.

#### Fix

`McpManager.kt`의 `ensureConnected`:

```kotlin
override suspend fun ensureConnected(serverName: String): Boolean {
    val status = statuses.getIfPresent(serverName)
    if (status == McpServerStatus.CONNECTED) return true
    // R332: PENDING + autoConnect=true만 첫 연결 허용. reconnection.enabled 플래그는
    // "degradation 이후 재연결" 용도이므로 첫 연결에는 적용하지 않는다.
    if (status == McpServerStatus.PENDING) {
        val server = servers.getIfPresent(serverName) ?: return false
        if (!server.autoConnect) return false
        logger.info { "MCP 서버 '$serverName' PENDING & autoConnect=true — 첫 연결 시도 (R332)" }
        return connect(serverName)
    }
    if (status != McpServerStatus.FAILED && status != McpServerStatus.DISCONNECTED) return false
    if (!reconnectionProperties.enabled) return false
    logger.info { "MCP 서버 온디맨드 재연결 시도: $serverName" }
    return connect(serverName)
}
```

**왜 `reconnectionProperties.enabled` 체크를 PENDING에 적용하지 않는가**: 그 플래그는 "일단 연결된 서버가 degradation된 뒤 자동 복구를 허용할 것인가"라는 운영 정책을 의미한다. PENDING은 degradation이 아니라 **첫 연결**이므로 의미가 다르다. `autoConnect=true`가 이미 "자동 관리"를 암시하므로 그것만으로 충분한 허용 신호.

#### 설계 메모

- 대안 1: `McpHealthPinger`의 PENDING 분기를 제거해 R173을 되돌린다. 즉 HealthPinger를 ensureConnected와 맞춤. 채택 안 함 — R173은 사용자 보호 의도가 있었고, autoConnect=true 서버 복구라는 real use case가 있다.
- 대안 2: `ensureConnected`가 PENDING 모두 허용. 채택 안 함 — `register()`만 호출한 서버는 사용자가 connect를 수동 제어하려는 의도이므로 health check가 침범하면 놀람.
- 채택: `autoConnect` 필드 기반 분기. 두 의도를 모두 존중하는 가장 narrow한 변경.
- 대안 1보다 더 디테일하게: health check는 이미 `McpHealthProperties.enabled`라는 자체 플래그를 가진다. 사용자가 health check 자체를 껐다면 PENDING 경로도 실행되지 않는다. 이는 추가 안전망.

#### User-visible Delta

- 시작 시 `initializeFromStore`에서 autoConnect=true 서버의 첫 연결이 transient 실패로 PENDING에 머물러도, health ping 주기에 자동으로 복구된다.
