#### Fix

`PlanExecuteStrategy.kt:synthesize` — `resultSummary` 빌더를 3가지 섹션으로 분류:

```kotlin
val resultSummary = results.joinToString("\n\n") { r ->
    val header = "[${r.tool}] ${r.description}"
    val body = when {
        !r.success -> "[실패] 이 단계는 실행에 실패했습니다. 답변 근거로 사용하지 마세요."
        r.output.isNullOrBlank() ->
            "[데이터 없음] 이 단계는 결과를 반환하지 않았습니다."
        else -> r.output
    }
    "$header\n$body"
}
```

**3가지 케이스**:
- **성공 + non-blank output**: 원본 output을 그대로 LLM에게 전달 (cross-source 합성 품질 유지)
- **성공 + blank output**: 명시적 "데이터 없음" 마커 (원래 빈 output이 LLM을 혼란하게 만드는 edge case 차단)
- **실패(`success=false`)**: 명시적 "[실패] 답변 근거로 사용하지 마세요" 마커. 원본 에러 문자열(`"Error: TOOL_ERROR"` 등) 노출 금지.

**핵심 원칙**:
1. LLM은 성공 source와 실패 source를 **명확히 구분**할 수 있다 → 부분 성공 케이스에서도 환각 없이 합성 가능
2. 내부 에러 표현은 LLM/사용자 응답 경로에서 **완전히 제거** → CLAUDE.md Gotcha #9 (HTTP/사용자 응답에 내부 에러 메시지 노출 금지) 계열 원칙 확장 적용
3. 이전 R308 fix("전부 실패는 우회")와 상호 보완 — R308은 no-answer 케이스를 차단하고, R333은 혼합 케이스의 환각 위험을 차단

#### 설계 메모

- **대안 1**: 실패 step을 `results.filter { it.success }`로 완전 제거. 채택 안 함 — LLM에게 "계획은 3단계였으나 한 단계가 실패했다"는 중요한 정보가 숨겨진다. 실패를 명시적으로 알리는 쪽이 더 정확한 합성을 유도한다.
- **대안 2**: 실패 step의 원본 `output`을 그대로 유지하되 `[실패]` prefix만 붙인다. 채택 안 함 — 원본 output은 `"Error: TOOL_ERROR"` 같은 내부 마커 문자열이고, 이를 LLM에 노출하는 것은 R308의 sanitization 원칙(`e.javaClass.simpleName`도 노출 금지)과 충돌한다.
- **채택**: 3-way when 분기. LLM에게는 **"실패했고 근거로 쓰지 말라"**는 신호만 전달하고 원본 에러는 서버 로그(`logger.warn`)로만 남긴다.
- §0.1 atlassian-mcp-server: 도구 발견/호출/응답 경로 전부 불변. synthesize는 LLM 프롬프트 내부 구조만 바뀌고 MCP wire protocol과 무관.
- §0.2 Redis cache: `CacheKeyBuilder`가 해싱하는 `systemPrompt`/`toolNames`/`model` 등은 불변. `resultSummary`는 userMessage 내부 콘텐츠이며 scopeFingerprint에 들어가지 않는다.
- §0.3 context: 대화/스레드 컨텍스트와 무관.
