#### 작업 배경

R333 scanner가 `cross_source_synthesis` axis 전환 라운드에서 3건을 발견했고, P1 HIGH(PlanExecute synthesize 실패 마커)를 R333에서 처리했다. 이번 R334는 동일 scanner의 defer 중 P2 MED #2(`mergeSignalMetadata` last-wins)를 이어서 처리한다.

**초기 분석과 재평가**:

| 항목 | 초기 해석 | 재평가 결과 |
|---|---|---|
| `answerMode`/`freshness`/`retrievedAt`/`grounded` last-wins | 버그로 추정 | **의도적 설계** — `ToolCallOrchestratorTest:517` `"Last merged signal should win metadata projection"` 테스트가 lock |
| `blockReason` last-wins | 버그로 추정 | **진짜 버그** — 보안 신호이므로 first-wins가 맞다 |

이 재평가는 buggy-vs-intentional을 구분하는 중요한 단계였다. 기존 테스트 lock은 "이 동작은 의도적이다"라는 강한 signal이므로 그 부분은 건드리지 않아야 한다. 대신 **차단 신호의 도메인 의미**("한 번 차단되면 계속 차단")를 근거로 `blockReason`만 예외적으로 first-wins로 변경한다.

**downstream 소비자 확인** (grep 결과):

- `AgentExecutionCoordinator.kt:346` — `if (result.metadata["blockReason"] != null) return` — 존재 여부로 분기
- `MicrometerAgentMetrics.kt:311` — `metadata["blockReason"]?.toString()?.isNotBlank() == true` — 존재 여부로 tag 생성
- `AgentMetrics.kt:374/415/426/487` — 동일
- `EvaluationMetricsHook.kt:28` — `hookContext.metadata["blockReason"]`를 metric label로 사용
- `ExecutionResultFinalizer.kt` — `blockReason`을 최종 응답 metadata로 복사

모든 소비자가 "첫 차단이 영구"라는 의미를 가정하고 있다. 구현이 그 의미를 배신하고 있었던 것이다.

#### Fix

`ToolCallOrchestrator.mergeSignalMetadata`의 `blockReason` 처리만 narrow하게 변경:

```kotlin
signal.blockReason?.let { newReason ->
    if (hookContext.metadata["blockReason"] == null) {
        hookContext.metadata["blockReason"] = newReason
    }
}
```

다른 4개 키(`answerMode`/`grounded`/`freshness`/`retrievedAt`)는 기존 last-wins 그대로:

```kotlin
signal.answerMode?.let { hookContext.metadata["answerMode"] = it }
signal.grounded?.let { hookContext.metadata["grounded"] = it }
signal.freshness?.let { hookContext.metadata["freshness"] = it }
signal.retrievedAt?.let { hookContext.metadata["retrievedAt"] = it }
```

**개별 signal 누적 보존**: `getOrCreateToolSignals(hookContext)`는 여전히 모든 signal을 `TOOL_SIGNALS_METADATA_KEY` 리스트에 누적한다. 즉 downstream이 전체 맥락(두 개 차단 사유가 모두 발생했다는 사실)을 필요로 하면 리스트를 읽으면 되고, flat key는 "첫 차단"만 노출한다.

#### 설계 메모

- **대안 1**: 5개 flat key를 전부 `"key_${toolName}"` per-tool namespace로 교체. 채택 안 함 — breaking change, downstream 소비자 전부 수정 필요, 기존 last-wins 테스트 재작성 필요.
- **대안 2**: 5개 key 전부 first-wins. 채택 안 함 — `ToolCallOrchestratorTest:517` 회귀 테스트 lock을 깨뜨림. last-wins는 의도적 설계.
- **대안 3**: `blockReason`만 "더 강한 차단 우선" 규칙(priority list 정의). 채택 안 함 — priority list 유지관리 비용 + 실제 use case에서 "첫 차단이 곧 확정"이 더 단순하고 도메인 의미와 맞음.
- **채택**: `blockReason`만 first-wins 예외 처리. 3줄 변경. 다른 경로 완전 불변.
- 주석에 "의도적 예외"임을 명시(`"R334 fix"` + "first-wins — 첫 차단 신호를 이후 tool이 덮어쓰지 못하게 한다").
