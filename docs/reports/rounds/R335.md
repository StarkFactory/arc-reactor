#### 작업 배경

R333에서 `cross_source_synthesis` axis로 전환하며 scanner가 3건(P1 HIGH 1 + P2 MED 2)을 발견했다. R333에서 P1 HIGH(synthesize 실패 마커), R334에서 P2 MED #2(mergeSignalMetadata blockReason first-wins)를 처리했고, R335가 마지막 P2 MED #3(incrementCounter atomic)을 처리해 **scanner batch를 완결**한다.

**실제 버그 분석**:

```kotlin
// BEFORE
private fun incrementCounter(context: ToolCallContext) {
    val current = (context.agentContext.metadata[COUNTER_KEY] as? Int) ?: 0
    context.agentContext.metadata[COUNTER_KEY] = current + 1
}
```

- 라인 1: `get(COUNTER_KEY)` 원자적 (ConcurrentHashMap)
- 라인 2: `+1` 계산
- 라인 3: `put(COUNTER_KEY, value)` 원자적 (ConcurrentHashMap)
- **문제**: 세 단계 전체가 원자적이지 않음. 두 코루틴 T1, T2가 각자 `current=0`을 읽고 각자 `put(..., 1)` → 최종 값 1. 실제로는 2가 되어야 함.

**왜 이걸 놓쳤는가**: 주석이 "원자적으로 증가시킨다"고 기재하고 있어 코드 리뷰에서 실제 구현이 주석을 따르는지 검증되지 않았다. scanner가 static 분석 대신 동시성 의미를 검토해 발견.

**영향도**: `toolSummaryCount`는 `ToolResponseSummarizer` 압축 메트릭의 집계 분모로 사용될 수 있다. 언더카운트는 `summarizer_compression_ratio` 등 관측 지표를 왜곡시켜 "요약 경로가 덜 활성화된 것처럼 보이는" 착시를 유발한다. user-visible는 아니지만 관측성 결함이므로 **silent observability loss**.

#### Fix

`incrementCounter`를 runtime type-check + `ConcurrentMap.compute` 기반으로 재작성:

```kotlin
private fun incrementCounter(context: ToolCallContext) {
    val metadata = context.agentContext.metadata
    @Suppress("UNCHECKED_CAST")
    val concurrent = metadata as? java.util.concurrent.ConcurrentMap<String, Any>
    if (concurrent != null) {
        concurrent.compute(COUNTER_KEY) { _, existing ->
            ((existing as? Int) ?: 0) + 1
        }
        return
    }
    // 테스트/커스텀 Map fallback
    synchronized(metadata) {
        val current = (metadata[COUNTER_KEY] as? Int) ?: 0
        metadata[COUNTER_KEY] = current + 1
    }
}
```

**설계 결정**:

- **대안 1**: `AtomicInteger`를 metadata value로 저장. 채택 안 함 — `metadata[COUNTER_KEY] as? Int`를 읽는 모든 소비자(테스트 + 잠재적 downstream)가 `AtomicInteger.get()`을 호출하도록 수정 필요. breaking change.
- **대안 2**: 전역 `synchronized(hook_instance)`. 채택 안 함 — global bottleneck 유발 + 여러 HookContext가 같은 hook instance를 공유하므로 서로 다른 request 간 lock contention.
- **대안 3**: `synchronized(metadata)` 전용. 채택 안 함 — metadata가 여러 다른 곳에서 lock monitor로 사용될 수 있어 deadlock 위험(제한적이나 불필요).
- **채택**: `ConcurrentMap.compute`. production 경로에서 원자적 read-modify-write를 한 연산으로 수행하며 lock granularity는 ConcurrentHashMap 내부 segment 단위라 성능 영향 최소. `MutableMap` 인터페이스 유지 + runtime 분기로 테스트 호환성 보존.
- 주석 수정: "원자적으로 증가시킨다" → 실제로 원자적이 되도록 구현을 바꾸고, 왜 이렇게 되어야 하는지 R335 배경을 KDoc에 명시.
