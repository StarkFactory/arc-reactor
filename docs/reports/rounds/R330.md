#### Fix 1 — `SHARED_HTTP_CLIENT_BUILDER` 제거 (P2 MED)

**이전 버그**: `companion object`에 단일 `java.net.http.HttpClient.Builder` 인스턴스를 보관한 뒤 `connectSse()`에서 `.connectTimeout(Duration.ofMillis(connectionTimeoutMs))`을 호출해 이 빌더를 **변경(mutate)**했다. `java.net.http.HttpClient.Builder`는 thread-safe 하지 않으며 setter는 동일 객체를 반환하면서 내부 상태를 덮어쓴다. 두 서버가 동시에 SSE 연결을 시도하면(애플리케이션 startup + `McpHealthPinger.reconnectScope`에서 병렬 reconnect) 동일 빌더를 동시에 mutate해 잘못된 timeout이 섞이거나 내부 상태가 오염될 수 있었다.

**원래 최적화 의도는 무효**: KDoc은 "매 연결 시도마다 새 HttpClient를 생성하면 TCP 소켓이 누적되어 TIME_WAIT 소켓 고갈이 발생한다"고 주장했으나, `HttpClientSseClientTransport`가 매번 `builder.build()`를 호출해 **새 `HttpClient` 인스턴스**를 만든다. 즉 공유 빌더로는 풀 재사용이 일어나지 않는다. 진정한 풀 공유를 원한다면 별도 라운드에서 build된 HttpClient 싱글턴을 도입해야 한다.

**수정**: `private val SHARED_HTTP_CLIENT_BUILDER`를 `private fun newHttpClientBuilder()`로 바꾸고, `connectSse()`에서 매번 새 빌더를 생성. 성능 비용은 사실상 0 (builder 생성은 가벼움, HttpClient 생성은 어차피 매번 발생).

#### Fix 2 — `handleConnectionError` client identity 체크 (P2 MED)

**이전 race**:
1. `connect()`가 mutex를 쥔 채 신규 클라이언트 C₂를 `clients.put()`하고 상태를 CONNECTED로 바꿈
2. 직전 reconnect 전 예전 클라이언트 C₁로 수행 중이던 도구 호출이 늦게 실패, 콜백 도착
3. 콜백 시그니처가 `(serverName)`뿐이라 `handleConnectionError`는 status=CONNECTED만 보고 **신규 C₂**를 `invalidate`하고 FAILED로 되돌림
4. 결과: 복구 직후 재실패 루프 + C₂ 핸들 leak + 도구 간헐 불가용

**수정**: 3-file 시그니처 확장 (identity 비교를 위한 최소 변경).

1. **`McpToolCallback`**: `onConnectionError: (() -> Unit)?` → `((failingClient: McpSyncClient) -> Unit)?`. `call()` catch 블록에서 `onConnectionError?.invoke(client)`로 자기 자신의 client ref 전달.
2. **`McpConnectionSupport`**: 생성자 `onConnectionError: (serverName: String) -> Unit` → `(serverName: String, failingClient: McpSyncClient) -> Unit`. `loadToolCallbacks`의 `McpToolCallback` 빌더에서 lambda 재작성.
3. **`McpManager.DefaultMcpManager`**: `handleConnectionError` 시그니처에 `failingClient: McpSyncClient` 추가. 내부에서 `clients.getIfPresent(serverName) !== failingClient`면 stale 이벤트로 판단하고 `debug` 로그 후 조용히 return. identity-동일한 경우에만 invalidate/close/FAILED 전이.

이 접근은 mutex보다 단순하며 정확한 방어선: `connect()`와 stale callback이 경합해도 항상 **현재 클라이언트**만 영향을 받는다.

#### 설계 메모

- identity 비교(`!==`)는 Mutex보다 가볍고 deadlock 위험이 없다. `handleConnectionError`는 non-suspend 함수여야 `onConnectionError`가 tool call 경로에서 자유롭게 호출될 수 있는 원 설계를 유지한다.
- `toolCallbacksCache`는 stale 콜백을 그대로 유지한다. 재연결 사이 윈도우에서 도구가 사라지면 UX가 나빠지고, 개별 도구 호출은 이미 `"Error: ..."` 문자열을 반환해 안전하다.
- R329 prompt §0.1 (atlassian-mcp-server 계약) 체크: SSE URL/스키마/도구 description 어느 것도 건드리지 않음. SSE transport 빌더의 `.clientBuilder()` 계약만 유지한다.
- Fix 3 (`reapplySecurityPolicy runBlocking`)은 다음 수동 라운드 후보. `McpManager` 인터페이스에 `suspend` 키워드 도입이 필요하여 호출자들(`ArcReactorAutoConfiguration`, admin controller 등) 전체를 함께 변환해야 한다.
